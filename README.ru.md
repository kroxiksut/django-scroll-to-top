`django-scroll-to-top` — переиспользуемое Django-приложение, которое добавляет
доступную и гибко настраиваемую кнопку прокрутки к началу страницы на сайт и в
стандартную Django Admin.

Сценарий для разработчика намеренно сделан минимальным:

1. Установить пакет и добавить его Django-приложение.
2. Выполнить миграции.
3. Добавить один template tag в базовый шаблон сайта.
4. Настраивать оформление и поведение через Django Admin.

Не требуются jQuery, внешний CDN, frontend-фреймворк или frontend-сборка.

## Зачем нужен этот пакет

Простую ссылку для прокрутки вверх написать легко. Полноценному
переиспользуемому компоненту также нужны:

- отдельное представление для компьютеров и мобильных устройств;
- клавиатурная навигация, фокус, контраст и уменьшение анимации;
- интеграция со стандартной Django Admin;
- безопасные пользовательские SVG-иконки;
- светлая и тёмная темы;
- учёт cookie-баннеров, чатов, закреплённой навигации и других fixed-элементов;
- скрытие по выбору пользователя без нарушения основной прокрутки;
- работа в проектах со строгой Content Security Policy;
- кэширование, локализация, упаковка и совместимость обновлений.

Пакет предоставляет эти возможности при минимальных изменениях шаблонов проекта.

## Возможности

- Один template tag для подключения на публичном сайте.
- Независимые конфигурации сайта и стандартной Django Admin.

- Черновики, публикация, откат и копирование конфигураций.
- Явные desktop-значения и наследование/переопределение каждого mobile-поля.
- Размещение сверху слева, сверху справа, снизу слева и снизу справа.
- Круглые, квадратные, скруглённые, вытянутые и текстовые шаблоны кнопок.
- Сплошная, контурная, мягкая, прозрачная, стеклянная и управляемая градиентная

## Установка

```console
python -m pip install django-scroll-to-top
python manage.py migrate
```

Добавьте приложение и включите необходимые области интеграции:

```python
INSTALLED_APPS = [
    # ...
    "django_scroll_to_top",
]

DJANGO_SCROLL_TO_TOP = {
    "SITE_ENABLED": True,
    "ADMIN_ENABLED": True,
}
```

Подключите package URLConf, чтобы публичный template tag мог загружать
версионированный same-origin stylesheet endpoint в режиме строгого CSP:

```python
from django.urls import include, path

urlpatterns = [
    # ...
    path(
        "scroll-to-top/",
        include(
            ("django_scroll_to_top.urls", "django_scroll_to_top"),
            namespace="django_scroll_to_top",
        ),
    ),
]
```

Пакет поддерживает и тестирует стандартные шаблоны Django Admin на заявленной
матрице совместимости Django. Интеграция использует документированные точки
расширения Django, сохраняет стандартный branding и не monkeypatch-ит закрытые
API Django.

Когда `DJANGO_SCROLL_TO_TOP["ADMIN_ENABLED"]` включён, размещайте
`"django_scroll_to_top"` раньше `"django.contrib.admin"` в `INSTALLED_APPS`,
чтобы footer-override шаблона `admin/base_site.html` выбирался обычным
механизмом поиска шаблонов Django.

Пользовательские экземпляры `AdminSite`, переопределённые базовые шаблоны Admin и
сторонние темы поддерживаются в режиме best effort и пока не входят в тестовую
матрицу совместимости. Проектам с такими интеграциями следует проверять их
локально и сообщать о несовместимостях, чтобы исследовать и исправлять их
совместно.

Для текущего стандартного footer-injection отдельное приложение интеграции с
админкой не требуется. Если в будущем оно понадобится, ему следует дать
собственный app label, а не перегружать label основного пакета.

### Recipe для custom AdminSite

Если проект переопределяет `admin/base_site.html` или использует собственное
дерево шаблонов `AdminSite`, сохраните обычные branding-блоки и добавьте tag
пакета в footer-block:

```django
{% extends "admin/base.html" %}
{% load i18n scroll_to_top %}

{% block branding %}
  {{ block.super }}
{% endblock %}

{% block footer %}
  {{ block.super }}
  {% scroll_to_top scope="admin" %}
{% endblock %}
```

Политика по умолчанию скрывает control на анонимных auth pages, например на
страницах login и password reset. Чтобы показывать его и там, установите
`DJANGO_SCROLL_TO_TOP["ADMIN_SHOW_ON_AUTH_PAGES"] = True`.

## Использование в шаблоне

Добавьте один tag ближе к концу общего базового шаблона:

```django
{% load scroll_to_top %}

<!-- Содержимое страницы -->

{% scroll_to_top %}
```

Tag получает опубликованную конфигурацию сайта, выводит выбранную безопасную
иконку и управляемый вариант шаблона, а также подключает ресурсы пакета. Обычные
визуальные параметры не передаются аргументами tag. Динамические цветовые и
размерные переменные доставляются через версионированный same-origin stylesheet,
а не через inline `style`, поэтому one-tag contract сохраняется без неявного
требования `unsafe-inline`.

Проект может переопределять документированные шаблоны пакета стандартным
механизмом Django template loaders.

## Модель конфигурации

Django settings отвечают только за установку и инфраструктуру. Обычное
оформление и поведение хранятся в конфигурации БД и редактируются через Django
Admin.

Области конфигурации:

| Область | Назначение |
| --- | --- |
| `site` | Публичные и прикладные страницы с template tag |
| `admin` | Страницы стандартной Django Admin |

Интеграция с Django Sites включена. Если установлен `django.contrib.sites`,
для текущего Site можно задать отдельный профиль с глобальной
fallback-конфигурацией. Пакет также продолжает работать без Sites Framework.
Настройки сайта и админки остаются независимыми, но их можно явно копировать.

Desktop-значения являются основными. Каждое mobile-поле либо наследует
desktop-значение, либо хранит явное переопределение. Административная форма
показывает эту связь, а не маскирует наследование пустыми значениями.

### Видимость и прокрутка

Каждая ревизия настраивает, когда кнопка появляется и куда прокручивает:

- режим порога видимости (`pixels`, `viewport` или `combined`) с `show_after_px`,
  `show_after_viewports` и минимумом `min_document_height_px`, чтобы на коротких
  страницах кнопка не появлялась;
- задержки показа/скрытия и направление видимости (`always`, `scroll_up_only`
  или `hide_on_scroll_down`);
- отключение на уровне страницы через атрибут `data-scroll-top="disabled"` на
  `<body>`;
- опциональный селектор цели прокрутки с вертикальным смещением и опциональный
  селектор фиксированной шапки, высота которой вычитается; при пустом или
  ненайденном селекторе — переход в начало документа;
- режим `smooth` или `instant` (при reduced-motion всегда мгновенный переход) с
  использованием нативного `window.scrollTo` без собственной анимационной петли.

### Визуальные шаблоны и оформление

Оформление строится из контролируемых CSS-классов пакета, без произвольных
шаблонов в БД:

- формы: `circle`, `square`, `rounded-square`, `pill`;
- заливки: `solid`, `outline`, `soft`, `ghost`, `glass` (полупрозрачная с
  fallback при отсутствии backdrop-blur) и `gradient` (два заданных цвета и угол);
- пресеты тени (`none`/`small`/`medium`/`large`), прозрачность, ширина границы,
  ширина/отступ фокус-кольца и величина backdrop-blur для варианта glass.

Неизвестные формы, заливки и тени откатываются к безопасным значениям, режим
forced-colors нейтрализует все варианты, а проект по-прежнему может переопределить
шаблон пакета стандартным механизмом Django.

## Размещение и коллизии с плавающими элементами

Для desktop и mobile доступны все четыре угла viewport, независимые отступы,
safe-area, ограниченный `z-index`, расстояние до препятствий и порядок резервных
углов.

Элемент сайта может объявить себя препятствием:

```html
<div data-scroll-top-obstacle>
  <!-- Cookie-баннер, launcher чата, закреплённое действие и т. п. -->
</div>
```

Администратор также может указать проверяемые CSS-селекторы. Браузерный runtime
измеряет видимые прямоугольники и применяет одну из политик:

- не учитывать препятствия;
- смещать кнопку вдоль выбранной стороны;
- пробовать настроенные резервные углы;
- скрывать кнопку, если безопасного положения нет.

Cookie- и chat-пакеты остаются необязательными и не становятся зависимостями
`django-scroll-to-top`.

### Опциональный адаптер препятствий

Для cookie-баннеров и других плавающих виджетов, которые трудно описать одним
статическим селектором, поставляется опциональный адаптер
`django_scroll_to_top/obstacle-adapter.js`. Тег `{% scroll_to_top %}` его никогда
не подключает — включайте только там, где он нужен:

```html
<script src="{% static 'django_scroll_to_top/obstacle-adapter.min.js' %}" defer></script>
<script>
  document.addEventListener("DOMContentLoaded", function () {
    // Общая регистрация: пометить селекторы и пересчитывать по событиям виджета.
    window.djsttObstacleAdapter.register({
      selectors: [".chat-widget", ".sticky-bottom-nav", ".toast-stack"],
      gap: 12,
      priority: 5,
      events: ["my-widget:open", "my-widget:close"]
    });

    // Либо использовать встроенный пресет django-cookies-152fz (панель + launcher).
    window.djsttObstacleAdapter.register(
      window.djsttObstacleAdapter.presets.djangoCookies152fz
    );
  });
</script>
```

Адаптер помечает подходящую разметку атрибутом `data-scroll-top-obstacle`
(в том числе вставленные позже элементы — например, компактный launcher,
появляющийся после закрытия баннера) и пробрасывает настроенные события
open/close/collapse в `window.djstt.refresh()`. Панель cookie-баннера и его
компактный launcher указаны отдельными селекторами, поэтому каждый измеряется по
своему bounding rectangle, пока виден. Содержимое cross-origin iframe никогда не
анализируется; пометка `<iframe>` заставляет движок считать препятствием сам
прямоугольник iframe.

Улучшения collision avoidance и theme-aware считаются progressive enhancement.
Базовый сценарий без JavaScript остаётся обычной ссылкой в начало страницы в
выбранном углу с консервативным встроенным preset.

## Иконки

Единый каталог поддерживает три источника:

- `builtin` — встроенные стартовые иконки Tabler;
- `developer` — иконки, зарегистрированные доверенным кодом проекта;
- `uploaded` — SVG, загруженные и одобренные через Django Admin.

Встроенный набор содержит подходящие стрелки, chevron, caret, круги, badge, bar
и квадратные варианты из [Tabler Icons](https://tabler.io/icons).

### Требования к цвету иконок

Кнопка берёт цвета из конфигурации в БД. Чтобы иконка подхватывала заданный цвет,
её SVG должен рисоваться через `currentColor`, а не жёстко заданным цветом — то
есть использовать `fill="currentColor"` (заливные иконки) или
`stroke="currentColor"` (контурные). Затем control применяет в порядке
приоритета:

1. **переопределение цвета иконки** (`icon_color` / `dark_icon_color`), если
   задано;
2. иначе **цвет переднего плана** (`foreground_color`), он же цвет подписи.

Многоцветные/оригинальные загруженные иконки сохраняют собственные цвета и
игнорируют эти поля (они не используют `currentColor`). Встроенные иконки Tabler
уже используют `currentColor`, поэтому перекрашиваются автоматически.

Tabler Icons распространяются по
[лицензии MIT](https://github.com/tabler/tabler-icons/blob/main/LICENSE). Она
разрешает широкое использование, изменение и распространение, включая
коммерческое, при сохранении copyright и текста лицензии. Релизные артефакты
содержат обязательное уведомление, источник и версию набора.

### Безопасность загружаемых SVG

Загруженный администратором SVG не выводится напрямую. Конвейер:

- разбирает SVG как XML;
- запрещает DTD, entities, скрипты, обработчики событий, внешние ресурсы,
  вложенные документы, небезопасные namespaces и чрезмерную сложность;
- разрешает только документированные графические элементы и атрибуты;
- нормализует геометрию и `viewBox`;
- хранит и выводит только очищенную версию;
- поддерживает перекрашивание совместимых иконок через `currentColor`;
- сохраняет безопасные исходные цвета только в явном многоцветном режиме.

Техническая очистка не предоставляет права использовать иконку. Для загруженных
иконок хранятся автор, источник, лицензия, copyright, attribution и подтверждение
администратора о праве проекта использовать и распространять файл.

Пакет не считает загруженный SVG свободным или открытым контентом лишь потому,
что файл принят санитайзером. Оператор сайта отвечает за наличие прав на
использование и распространение, корректность полей attribution и выгрузку отчёта
по используемым иконкам, если он нужен для конкретного развёртывания.

## Темы и цвета

Заданные администратором цвета доставляются через генерируемый пакетом
версионированный same-origin stylesheet: произвольный CSS не хранится, а
`unsafe-inline` не требуется. Stylesheet использует ту же разрешённую
конфигурацию Site/profile, что и renderer, и работает без JavaScript.

Поддерживаются ручной режим и наследование. Конфигурация стандартной Django
Admin может наследовать поддерживаемые CSS variables админки, реагировать на
светлую/тёмную тему и иметь безопасные fallback. Встроенный режим
`inherit_admin_theme` сначала использует наблюдаемые CSS variables Django Admin,
при их отсутствии переходит на нейтральный preset и дополнительно даёт namespaced
adapter hooks `--dstt-admin-*` для сторонних admin themes.

В остальном цвета задаются вручную по ревизиям. (Прежний браузерный анализ цветов
страницы удалён как ненужный.)

## Скрытие пользователем

Постоянное скрытие отделяется от временной runtime-видимости и от
административного флага включения. Каждая ревизия настраивает скрытие:

- `allow_user_dismissal` выводит видимый элемент закрытия;
- механизм хранения — `local`, `session`, функциональный `cookie` или `none`
  (только в памяти). Режим по умолчанию `local` ничего не сохраняет, пока
  посетитель действительно не скрыл кнопку;
- срок — либо `persistent` (до смены токена конфигурации ревизии или
  `dismissal_version`), либо по дням через `dismissal_days` с самоочисткой по
  истечении;
- `dismissal_requires_confirmation` запрашивает подтверждение перед скрытием;
- `dismissal_version` — явный переключатель для повторного показа кнопки.

Ключи хранения namespaced по области, сайту, токену конфигурации и версии
скрытия; доступ к хранилищу устойчив к запрету/недоступности и не ломает
прокрутку. Кнопку можно восстановить программно через `window.djstt.restore()`.
Тройной клик остаётся экспериментальным: первый обычный клик запускает прокрутку
и может сразу скрыть кнопку, поэтому по умолчанию он не включён.

## Доступность

Компонент ориентирован на WCAG 2.2 AA в пределах своей области:

- доступное имя не зависит от иконки или tooltip;
- управление клавиатурой и видимый фокус;
- достаточный размер области нажатия;
- проверка контраста normal, hover, active и focus;
- поддержка `prefers-reduced-motion`;
- работа в forced-colors/high-contrast;
- поддержка масштабирования и RTL;
- отсутствие keyboard trap и неожиданного перемещения фокуса;
- декоративный SVG скрыт от accessibility API.

Структурный контракт выше реализован и покрыт тестами: переводимое доступное имя
у кнопки и у элемента закрытия, минимальный целевой размер 24x24 CSS px,
fallback-ссылка без JavaScript, отсутствие положительного tabindex, видимые
focus-visible-обводки, обработка forced-colors и reduced-motion и безопасные для
RTL логические свойства. Полный аудит WCAG 2.2 AA и проверка zoom (200%/400%) в
реальных браузерах запланированы как отдельные этапы стабилизации (см.
[Roadmap](#roadmap)).

## Безопасность, приватность и CSP

- Через административные формы нельзя хранить произвольные HTML, JavaScript или CSS.
- Production никогда не выводит неочищенный исходный SVG.
- Telemetry и внешние сетевые обращения по умолчанию отсутствуют.
- Строгий CSP не требует неявно `unsafe-inline`.
- Ошибки browser storage обрабатываются безопасно.
- Cookie- и DB-хранение скрытия необязательны и документируются отдельно, если
  включены.

О поддерживаемых версиях и о том, как сообщить об уязвимости, см.
[SECURITY.ru.md](SECURITY.ru.md).

### Минимальная CSP-конфигурация

По умолчанию `DJANGO_SCROLL_TO_TOP["CSP_MODE"]` равен `"external"`. В этом
режиме компонент использует same-origin теги `<link>` и `<script>` и совместим
с политикой вида:

```text
Content-Security-Policy:
  default-src 'self';
  style-src 'self';
  script-src 'self';
```

Если проект использует nonce-режим для script, задайте
`DJANGO_SCROLL_TO_TOP["CSP_MODE"] = "nonce"` и передавайте `csp_nonce` в
template context или `request.csp_nonce`. Пакет добавит этот nonce во внешний
script tag, а стили по-прежнему будут приходить через same-origin stylesheet
endpoint.

## Runtime-события

Browser runtime использует одну документированную глобальную точку входа
`window.djstt` с помощниками `init(root?)`, `refresh(root?)` и `destroy(root?)`
для progressive enhancement и интеграции с partial navigation.

Каждый component root отправляет namespaced DOM-события:

- `djstt:show`
- `djstt:hide`
- `djstt:scroll-start`
- `djstt:scroll-end`
- `djstt:dismiss`

Для HTMX-подобной замены фрагмента можно вызвать
`window.djstt.init(fragmentRoot)`. Для Turbo и похожих full-page navigation
можно вызвать `window.djstt.refresh()`.

## Точки расширения

Опциональные хуки настраиваются в `DJANGO_SCROLL_TO_TOP` как dotted path или
callable; сбойный хук не ломает рендеринг:

- `SITE_ID_RESOLVER(request) -> int | None` — определить текущий Site id без
  зависимости от `django.contrib.sites`;
- `PROFILE_RESOLVER(scope, site_id) -> ScrollTopProfile | None` — переопределить
  выбор профиля/ревизии; `None` возвращает к встроенному разрешению;
- `OBSTACLE_SELECTORS() -> list[str]` — добавить obstacle-селекторы в каждый
  отрисованный контрол.

Иконки разработчика регистрируются через `register_developer_icon(...)`.
Стабильная публичная поверхность — `window.djstt`
(`init`/`refresh`/`destroy`, `version`, `dismiss`/`restore`/`debug`),
namespaced DOM-события `djstt:*`, атрибуты `data-dstt-*` на обёртке контрола,
маркер `data-scroll-top-obstacle` и шаблон
`django_scroll_to_top/scroll_to_top.html`. Внутренние service/model API не
являются публичным контрактом.

## Диагностика

Системные проверки сообщают о типичных ошибках конфигурации со стабильными id и
понятными подсказками:

- `dstt.W001`/`W002`: миграции нельзя проинспектировать или они не применены.
- `dstt.W003`: неподдерживаемый `CSP_MODE`.
- `dstt.W004`: URLConf пакета отсутствует при включённом рендеринге сайта.
- `dstt.W005`: `django_scroll_to_top` стоит после `django.contrib.admin`.
- `dstt.W006`: неподдерживаемый `DEFAULT_COLLISION_POLICY`.
- `dstt.W007`: `DJANGO_SCROLL_TO_TOP` не словарь или содержит неизвестные ключи.
- `dstt.W008`: `ADMIN_ENABLED` включён, но `django.contrib.admin` не установлен.
- `dstt.W009`: `SITES_FRAMEWORK_ENABLED` включён, но `django.contrib.sites` не
  установлен.
- `dstt.W010`: отсутствуют упакованные шаблоны или минифицированная статика.

Две management-команды помогают диагностировать развёртывание:

```console
python manage.py scroll_to_top_diagnose        # разрешённая конфигурация по областям, без секретов
python manage.py scroll_to_top_check_contrast  # ненулевой код, если ревизия не проходит контраст
```

## Совместимость

Матрица охватывает Django 4.2 LTS, 5.x и 6.x с версиями Python, которые
поддерживает выбранный релиз Django. Поддержка ветки Django 4 ограничивается
именно 4.2 LTS, а не всей линией 4.x. Матрица закреплена в `pyproject.toml` и
проверяется в CI (`tox.ini` её отражает).

Базовые runtime-зависимости ограничены Django.

## Обратная связь

Это ранний релиз (`0.x` beta), поэтому отзывы из реальной эксплуатации особенно
ценны. Сообщения об ошибках и предложения приветствуются через
[GitHub issues](https://github.com/kroxiksut/django-scroll-to-top/issues),
особенно по темам:

- **Интеграция с админкой** — пакет протестирован на стандартных шаблонах Django
  Admin. Пользовательские экземпляры `AdminSite`, переопределённые базовые
  шаблоны админки и сторонние admin-темы поддерживаются в режиме best-effort и
  пока не входят в тестовую матрицу совместимости; пожалуйста, сообщайте, что
  работает, а что нет.
- **Поведение фронтенда** — обход коллизий с реальными cookie-баннерами,
  launcher'ами чатов, sticky-навигацией и стеками toast; размещение при разных
  размерах viewport и safe-area; работа при строгом CSP; наследование темы со
  сторонними admin-темами; слои partial navigation (HTMX, Turbo).
- **Доступность в реальных браузерах** — клавиатура, фокус, контраст,
  forced-colors, reduced motion, RTL и zoom (200%/400%); незакрытые аудиты см. в
  разделе [Roadmap](#roadmap).

О том, как составить полезный отчёт, см. [CONTRIBUTING.ru.md](CONTRIBUTING.ru.md).
