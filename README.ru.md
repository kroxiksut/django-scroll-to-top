> дистрибутивы. Пока версия остаётся в серии `0.x`, публичный API может меняться
> между минорными релизами до `1.0.0`.

`django-scroll-to-top` — переиспользуемое Django-приложение, которое добавляет
доступную и гибко настраиваемую кнопку прокрутки к началу страницы на сайт и в
стандартную Django Admin.

Сценарий для разработчика намеренно сделан минимальным:

1. Установить пакет и добавить его Django-приложение.
2. Выполнить миграции.
3. Добавить один тег шаблона в базовый шаблон сайта.
4. Настраивать оформление и поведение через Django Admin.

Не требуются jQuery, внешний CDN, фронтенд-фреймворк или сборка фронтенда.

> **Большинству проектов достаточно быстрого старта: установка → миграции →
> один тег шаблона.** Встроенные значения по умолчанию готовы к продакшену, в
> админку заходить не обязательно. Всё начиная с
> [модели конфигурации](#модель-конфигурации) — опционально и подробнее
> раскрыто в [`docs/`](docs/README.ru.md).

## Зачем нужен этот пакет

Простую ссылку для прокрутки вверх написать легко. Полноценному
переиспользуемому компоненту также нужны:

- отдельное представление для компьютеров и мобильных устройств;
- клавиатурная навигация, фокус, контраст и уменьшение анимации;
- интеграция со стандартной Django Admin;
- безопасные пользовательские SVG-иконки;
- светлая и тёмная темы;
- учёт cookie-баннеров, чатов, закреплённой навигации и других фиксированных элементов;
- скрытие по выбору пользователя без нарушения основной прокрутки;
- работа в проектах со строгой Content Security Policy;
- кэширование, локализация, упаковка и совместимость обновлений.

Пакет предоставляет эти возможности при минимальных изменениях шаблонов проекта.

## Возможности

- Один тег шаблона для подключения на публичном сайте.
- Независимые конфигурации сайта и стандартной Django Admin.
- Хранение конфигурации в БД и управление через Django Admin.
- Черновики, публикация, откат и копирование конфигураций.
- Явные значения для компьютеров и наследование/переопределение каждого мобильного поля.
- Размещение сверху слева, сверху справа, снизу слева и снизу справа.
- Круглые, квадратные, скруглённые, вытянутые и текстовые шаблоны кнопок.
- Сплошная, контурная, мягкая, прозрачная, стеклянная и управляемая градиентная
  заливка.
- Настройка цветов, границ, теней, фокус-кольца, отступов и размеров.
- Встроенные светлые/тёмные предустановки и наследование темы.
- Интерактивный предпросмотр для компьютеров и мобильных на основе боевого рендерера.
- Встроенные стартовые иконки Tabler, регистрация разработчиком и загрузка
  администратором.
- Безопасная загрузка, очистка, предпросмотр, перекрашивание и лицензирование SVG.
- Пороги появления, определение короткой страницы, учёт фиксированной шапки и
  необязательный селектор цели.
- Нативная плавная прокрутка с поддержкой `prefers-reduced-motion`.
- Обход коллизий с баннерами, кнопками вызова, чатами, контейнерами всплывающих
  уведомлений (toast), закреплённой навигацией и другими плавающими элементами.
- Локальное/сессионное скрытие пользователем со сроком, версией сброса и
  возможностью восстановления.
- Django i18n: английские исходные строки и русский как первый встроенный перевод.

## Установка

```console
python -m pip install django-scroll-to-top
python manage.py migrate
```

Добавьте приложение и включите необходимые области интеграции:

```python
INSTALLED_APPS = [
    # ...
    "django_scroll_to_top",
]

DJANGO_SCROLL_TO_TOP = {
    "SITE_ENABLED": True,
    "ADMIN_ENABLED": True,
}
```

Подключите URL-конфигурацию пакета, чтобы публичный тег шаблона мог загружать
версионированную таблицу стилей с того же источника в режиме строгого CSP:

```python
from django.urls import include, path

urlpatterns = [
    # ...
    path(
        "scroll-to-top/",
        include(
            ("django_scroll_to_top.urls", "django_scroll_to_top"),
            namespace="django_scroll_to_top",
        ),
    ),
]
```

> **Первый запуск.** После `migrate` элемент уже отрисовывается из безопасных
> встроенных значений, поэтому три секции админки поначалу пусты — это ожидаемо,
> а не ошибка. Чтобы настроить своё, нажмите **«Создать стартовую конфигурацию»**
> в списке профилей (один клик создаёт и публикует профиль с ревизией по
> умолчанию для каждой области) либо создайте профиль, добавьте ревизию и
> запустите **«Опубликовать выбранную ревизию»**. Дальше конкретный внешний вид и
> поведение кнопки настраиваются правкой опубликованной **ревизии** (форма, цвета,
> размеры, иконка, размещение, видимость), а не профиля. Меню из трёх пунктов
> можно свернуть переключателем в левой боковой панели админки. Подробнее —
> [operations-admin](docs/ru/operations-admin.md).

Пакет поддерживает и тестирует стандартные шаблоны Django Admin на заявленной
матрице совместимости Django. Интеграция использует документированные точки
расширения Django, сохраняет стандартное оформление и не подменяет закрытые API
Django (без monkeypatching).

Когда `DJANGO_SCROLL_TO_TOP["ADMIN_ENABLED"]` включён, размещайте
`"django_scroll_to_top"` раньше `"django.contrib.admin"` в `INSTALLED_APPS`,
чтобы переопределение шаблона `admin/base_site.html` в нижнем блоке (`footer`)
выбиралось обычным механизмом поиска шаблонов Django.

Пользовательские экземпляры `AdminSite`, переопределённые базовые шаблоны Admin и
сторонние темы поддерживаются по мере возможностей и пока не входят в тестовую
матрицу совместимости. Проектам с такими интеграциями следует проверять их
локально и сообщать о несовместимостях, чтобы исследовать и исправлять их
совместно.

Для текущей стандартной вставки в `footer` отдельное приложение интеграции с
админкой не требуется. Если в будущем оно понадобится, ему следует дать
собственный ярлык приложения (`app label`), а не перегружать ярлык основного
пакета.

### Рецепт для пользовательского `AdminSite`

Если проект переопределяет `admin/base_site.html` или использует собственное
дерево шаблонов `AdminSite`, сохраните обычные блоки оформления (`branding`) и
добавьте тег пакета в блок `footer`:

```django
{% extends "admin/base.html" %}
{% load i18n scroll_to_top %}

{% block branding %}
  {{ block.super }}
{% endblock %}

{% block footer %}
  {{ block.super }}
  {% scroll_to_top scope="admin" %}
{% endblock %}
```

Политика по умолчанию скрывает кнопку на анонимных страницах аутентификации —
например, на страницах входа и сброса пароля. Чтобы показывать её и там,
установите `DJANGO_SCROLL_TO_TOP["ADMIN_SHOW_ON_AUTH_PAGES"] = True`.

## Использование в шаблоне

Добавьте один тег ближе к концу общего базового шаблона:

```django
{% load scroll_to_top %}

<!-- Содержимое страницы -->

{% scroll_to_top %}
```

Тег получает опубликованную конфигурацию сайта, выводит выбранную безопасную
иконку и управляемый вариант шаблона, а также подключает ресурсы пакета. Обычные
визуальные параметры не передаются аргументами тега. Динамические цветовые и
размерные переменные доставляются через версионированную таблицу стилей с того же
источника, а не через встроенный атрибут `style`, поэтому контракт «один тег»
сохраняется без неявного требования `unsafe-inline`.

Проект может переопределять документированные шаблоны пакета стандартным
механизмом загрузчиков шаблонов Django.

## Модель конфигурации

Настройки Django отвечают только за установку и инфраструктуру. Обычное
оформление и поведение хранятся в конфигурации БД и редактируются через Django
Admin.

Области конфигурации:

| Область | Назначение |
| --- | --- |
| `site` | Публичные и прикладные страницы с тегом шаблона |
| `admin` | Страницы стандартной Django Admin |

Интеграция с Django Sites включена. Если установлен `django.contrib.sites`,
для текущего `Site` можно задать отдельный профиль с глобальной запасной
конфигурацией. Пакет также продолжает работать без Sites Framework.
Настройки сайта и админки остаются независимыми, но их можно явно копировать.

Значения для компьютеров являются основными. Каждое мобильное поле либо наследует
значение для компьютеров, либо хранит явное переопределение. Административная
форма показывает эту связь, а не маскирует наследование пустыми значениями.

### Видимость и прокрутка

Каждая ревизия настраивает, когда кнопка появляется и куда прокручивает:

- режим порога видимости (`pixels`, `viewport` или `combined`) с `show_after_px`,
  `show_after_viewports` и минимумом `min_document_height_px`, чтобы на коротких
  страницах кнопка не появлялась;
- задержки показа/скрытия и направление видимости (`always`, `scroll_up_only`
  или `hide_on_scroll_down`);
- отключение на уровне страницы через атрибут `data-scroll-top="disabled"` на
  `<body>`;
- опциональный селектор цели прокрутки с вертикальным смещением и опциональный
  селектор фиксированной шапки, высота которой вычитается; при пустом или
  ненайденном селекторе — переход в начало документа;
- режим `smooth` или `instant` (при `prefers-reduced-motion` всегда мгновенный
  переход) с использованием нативного `window.scrollTo` без собственной
  анимационной петли.

### Визуальные шаблоны и оформление

Оформление строится из контролируемых CSS-классов пакета, без произвольных
шаблонов в БД:

- формы: `circle`, `square`, `rounded-square`, `pill`;
- заливки: `solid`, `outline`, `soft`, `ghost`, `glass` (полупрозрачная с
  запасным вариантом при отсутствии backdrop-blur) и `gradient` (два заданных
  цвета и угол);
- предустановки тени (`none`/`small`/`medium`/`large`), прозрачность, ширина
  границы, ширина/отступ фокус-кольца и величина backdrop-blur для варианта
  `glass`.

Шесть вариантов заливки (показаны на круглой кнопке по умолчанию):

## Размещение и коллизии с плавающими элементами

Для компьютеров и мобильных доступны все четыре угла области просмотра,
независимые отступы, безопасная область (safe-area), ограниченный `z-index`,
расстояние до препятствий и порядок резервных углов.

Элемент сайта может объявить себя препятствием:

```html
<div data-scroll-top-obstacle>
  <!-- Cookie-баннер, кнопка вызова чата, закреплённое действие и т. п. -->
</div>
```

Администратор также может указать проверяемые CSS-селекторы. Браузерный слой
измеряет видимые прямоугольники и применяет одну из политик:

- не учитывать препятствия;
- смещать кнопку вдоль выбранной стороны;
- пробовать настроенные резервные углы;
- скрывать кнопку, если безопасного положения нет.

Cookie- и чат-пакеты остаются необязательными и не становятся зависимостями
`django-scroll-to-top`.

### Опциональный адаптер препятствий

Для cookie-баннеров и других плавающих виджетов, которые трудно описать одним
статическим селектором, поставляется опциональный адаптер
`django_scroll_to_top/obstacle-adapter.js`. Тег `{% scroll_to_top %}` его никогда
не подключает — включайте только там, где он нужен:

```html
<script src="{% static 'django_scroll_to_top/obstacle-adapter.min.js' %}" defer></script>
<script>
  document.addEventListener("DOMContentLoaded", function () {
    // Общая регистрация: пометить селекторы и пересчитывать по событиям виджета.
    window.djsttObstacleAdapter.register({
      selectors: [".chat-widget", ".sticky-bottom-nav", ".toast-stack"],
      gap: 12,
      priority: 5,
      events: ["my-widget:open", "my-widget:close"]
    });

    // Либо использовать встроенную предустановку django-cookies-152fz (панель + кнопка вызова).
    window.djsttObstacleAdapter.register(
      window.djsttObstacleAdapter.presets.djangoCookies152fz
    );
  });
</script>
```

Адаптер помечает подходящую разметку атрибутом `data-scroll-top-obstacle`
(в том числе вставленные позже элементы — например, компактную кнопку вызова,
появляющуюся после закрытия баннера) и пробрасывает настроенные события
открытия/закрытия/сворачивания в `window.djstt.refresh()`. Панель cookie-баннера
и его компактная кнопка вызова указаны отдельными селекторами, поэтому каждая
измеряется по своему ограничивающему прямоугольнику, пока видна. Содержимое
iframe другого источника никогда не анализируется; пометка `<iframe>` заставляет
движок считать препятствием сам прямоугольник iframe.

Улучшения обхода коллизий и учёта темы считаются прогрессивным улучшением.
Базовый сценарий без JavaScript остаётся обычной ссылкой в начало страницы в
выбранном углу с консервативной встроенной предустановкой.

## Иконки

Единый каталог поддерживает три источника:

- `builtin` — встроенные стартовые иконки Tabler;
- `developer` — иконки, зарегистрированные доверенным кодом проекта;
- `uploaded` — SVG, загруженные и одобренные через Django Admin.

Встроенный набор содержит подходящие стрелки, шевроны, каретки, круги, значки,
полоски и квадратные варианты из [Tabler Icons](https://tabler.io/icons).

### Требования к цвету иконок

Кнопка берёт цвета из конфигурации в БД. Чтобы иконка подхватывала заданный цвет,
её SVG должен рисоваться через `currentColor`, а не жёстко заданным цветом — то
есть использовать `fill="currentColor"` (заливные иконки) или
`stroke="currentColor"` (контурные). Затем кнопка применяет в порядке
приоритета:

1. **переопределение цвета иконки** (`icon_color` / `dark_icon_color`), если
   задано;
2. иначе **цвет переднего плана** (`foreground_color`), он же цвет подписи.

Многоцветные/оригинальные загруженные иконки сохраняют собственные цвета и
игнорируют эти поля (они не используют `currentColor`). Встроенные иконки Tabler
уже используют `currentColor`, поэтому перекрашиваются автоматически.

Tabler Icons распространяются по
[лицензии MIT](https://github.com/tabler/tabler-icons/blob/main/LICENSE). Она
разрешает широкое использование, изменение и распространение, включая
коммерческое, при сохранении уведомления об авторских правах и текста лицензии.
Релизные артефакты содержат обязательное уведомление, источник и версию набора.

### Безопасность загружаемых SVG

Загруженный администратором SVG не выводится напрямую. Конвейер:

- разбирает SVG как XML;
- запрещает DTD, сущности (entities), скрипты, обработчики событий, внешние
  ресурсы, вложенные документы и небезопасные пространства имён;
- разрешает только документированные графические элементы и атрибуты;
- применяет явные лимиты сложности (см. ниже);
- нормализует геометрию и `viewBox`;
- хранит и выводит только очищенную версию;
- поддерживает перекрашивание совместимых иконок через `currentColor`;
- сохраняет безопасные исходные цвета только в явном многоцветном режиме.

Лимиты сложности заданы явными константами в `icons/sanitizer.py`; всё, что
превышает лимит или использует элемент/атрибут вне белого списка, отклоняется, а
не обрезается:

| Лимит | Значение |
| --- | --- |
| Максимальный размер файла | 100 КБ (`MAX_SVG_BYTES = 100_000`) |
| Максимальное число элементов (узлов) | 128 (`MAX_ELEMENT_COUNT`) |
| Максимальная глубина вложенности | 8 (`MAX_XML_DEPTH`) |
| Максимум геометрии на атрибут (`d`, `points`) | 20 000 символов (`MAX_PATH_DATA_LENGTH`) |
| Элементы и атрибуты | строгий белый список (без `style`, `class`, `id`, `href`) |

Техническая очистка не предоставляет права использовать иконку. Для загруженных
иконок хранятся автор, источник, лицензия, сведения об авторских правах, указание
авторства и подтверждение администратора о праве проекта использовать и
распространять файл.

Пакет не считает загруженный SVG свободным или открытым контентом лишь потому,
что файл прошёл очистку. Оператор сайта отвечает за наличие прав на использование
и распространение, корректность полей указания авторства и выгрузку отчёта по
используемым иконкам, если он нужен для конкретного развёртывания.

## Темы и цвета

Заданные администратором цвета доставляются через генерируемую пакетом
версионированную таблицу стилей с того же источника: произвольный CSS не хранится,
а `unsafe-inline` не требуется. Таблица стилей использует ту же разрешённую
конфигурацию `Site`/профиля, что и рендерер, и работает без JavaScript.

Поддерживаются ручной режим и наследование. Конфигурация стандартной Django
Admin может наследовать поддерживаемые CSS-переменные админки, реагировать на
светлую/тёмную тему и иметь безопасные запасные значения. Встроенный режим
`inherit_admin_theme` сначала использует наблюдаемые CSS-переменные Django Admin,
при их отсутствии переходит на нейтральную предустановку и дополнительно даёт
хуки адаптера с пространством имён `--dstt-admin-*` для сторонних тем админки.

В остальном цвета задаются вручную по ревизиям. (Прежний браузерный анализ цветов
страницы удалён как ненужный.)

## Скрытие пользователем

Постоянное скрытие отделяется от временной видимости во время работы и от
административного флага включения. Каждая ревизия настраивает скрытие:

- `allow_user_dismissal` выводит видимый элемент закрытия;
- механизм хранения — `local`, `session`, функциональный `cookie` или `none`
  (только в памяти). Режим по умолчанию `local` ничего не сохраняет, пока
  посетитель действительно не скрыл кнопку;
- срок — либо `persistent` (до смены токена конфигурации ревизии или
  `dismissal_version`), либо по дням через `dismissal_days` с самоочисткой по
  истечении;
- `dismissal_requires_confirmation` запрашивает подтверждение перед скрытием;
- `dismissal_version` — явный переключатель для повторного показа кнопки.

Ключи хранения снабжены пространством имён по области, сайту, токену конфигурации
и версии скрытия; доступ к хранилищу устойчив к запрету/недоступности и не ломает
прокрутку. Кнопку можно восстановить программно через `window.djstt.restore()`.
Тройной клик остаётся экспериментальным: первый обычный клик запускает прокрутку
и может сразу скрыть кнопку, поэтому по умолчанию он не включён.

## Доступность

Компонент спроектирован под соответствие WCAG 2.2 AA в пределах своей области.
Полный независимый аудит и проверка масштабирования 200%/400% в реальных
браузерах ещё запланированы (см. [планы развития](#планы-развития)); описанный
ниже структурный контракт реализован и покрыт тестами:

- доступное имя не зависит от иконки или всплывающей подсказки;
- управление клавиатурой и видимый фокус;
- достаточный размер области нажатия;
- проверка контраста normal, hover, active и focus;
- поддержка `prefers-reduced-motion`;
- работа в forced-colors/high-contrast;
- поддержка масштабирования и RTL;
- отсутствие ловушки фокуса с клавиатуры и неожиданного перемещения фокуса;
- декоративный SVG скрыт от API доступности.

Структурный контракт выше реализован и покрыт тестами: переводимое доступное имя
у кнопки и у элемента закрытия, минимальный целевой размер 24×24 CSS px,
запасная ссылка без JavaScript, отсутствие положительного `tabindex`, видимые
обводки фокуса (`:focus-visible`), обработка forced-colors и reduced-motion и
безопасные для RTL логические свойства. Полный аудит WCAG 2.2 AA и проверка
масштабирования (200%/400%) в реальных браузерах запланированы как отдельные
этапы стабилизации (см. [планы развития](#планы-развития)).

## Безопасность, приватность и CSP

- Через административные формы нельзя хранить произвольные HTML, JavaScript или CSS.
- В боевом режиме никогда не выводится неочищенный исходный SVG.
- Телеметрия и внешние сетевые обращения по умолчанию отсутствуют.
- Строгий CSP не требует неявно `unsafe-inline`.
- Ошибки браузерного хранилища обрабатываются безопасно.
- Cookie- и хранение скрытия в БД необязательны и документируются отдельно, если
  включены.

О поддерживаемых версиях и о том, как сообщить об уязвимости, см.
[SECURITY.ru.md](SECURITY.ru.md).

### Минимальная CSP-конфигурация

По умолчанию `DJANGO_SCROLL_TO_TOP["CSP_MODE"]` равен `"external"`. В этом
режиме компонент использует теги `<link>` и `<script>` с того же источника и
совместим с политикой вида:

```text
Content-Security-Policy:
  default-src 'self';
  style-src 'self';
  script-src 'self';
```

Если проект использует nonce-режим для скриптов, задайте
`DJANGO_SCROLL_TO_TOP["CSP_MODE"] = "nonce"` и передавайте `csp_nonce` в
контекст шаблона или `request.csp_nonce`. Пакет добавит этот nonce во внешний
тег `<script>`, а стили по-прежнему будут приходить через таблицу стилей с того
же источника.

## События браузерного слоя

Браузерный слой использует одну документированную глобальную точку входа
`window.djstt` с помощниками `init(root?)`, `refresh(root?)` и `destroy(root?)`
для прогрессивного улучшения и интеграции с частичной навигацией.

Каждый корень компонента отправляет DOM-события с пространством имён:

- `djstt:show`
- `djstt:hide`
- `djstt:scroll-start`
- `djstt:scroll-end`
- `djstt:dismiss`

Для HTMX-подобной замены фрагмента можно вызвать
`window.djstt.init(fragmentRoot)`. Для Turbo и похожей полностраничной навигации
можно вызвать `window.djstt.refresh()`.

## Точки расширения

Опциональные хуки настраиваются в `DJANGO_SCROLL_TO_TOP` как путь через точку или
вызываемый объект; сбойный хук не ломает рендеринг:

- `SITE_ID_RESOLVER(request) -> int | None` — определить текущий идентификатор
  `Site` без зависимости от `django.contrib.sites`;
- `PROFILE_RESOLVER(scope, site_id) -> ScrollTopProfile | None` — переопределить
  выбор профиля/ревизии; `None` возвращает к встроенному разрешению;
- `OBSTACLE_SELECTORS() -> list[str]` — добавить селекторы препятствий в каждую
  отрисованную кнопку.

Иконки разработчика регистрируются через `register_developer_icon(...)`.
Стабильная публичная поверхность — `window.djstt`
(`init`/`refresh`/`destroy`, `version`, `dismiss`/`restore`/`debug`),
DOM-события `djstt:*` с пространством имён, атрибуты `data-dstt-*` на обёртке
кнопки, маркер `data-scroll-top-obstacle` и шаблон
`django_scroll_to_top/scroll_to_top.html`. Внутренние API сервисов и моделей не
являются публичным контрактом.

## Диагностика

Системные проверки сообщают о типичных ошибках конфигурации со стабильными id и
понятными подсказками:

- `dstt.W001`/`W002`: миграции нельзя проинспектировать или они не применены.
- `dstt.W003`: неподдерживаемый `CSP_MODE`.
- `dstt.W004`: URL-конфигурация пакета отсутствует при включённом рендеринге сайта.
- `dstt.W005`: `django_scroll_to_top` стоит после `django.contrib.admin`.
- `dstt.W006`: неподдерживаемый `DEFAULT_COLLISION_POLICY`.
- `dstt.W007`: `DJANGO_SCROLL_TO_TOP` не словарь или содержит неизвестные ключи.
- `dstt.W008`: `ADMIN_ENABLED` включён, но `django.contrib.admin` не установлен.
- `dstt.W009`: `SITES_FRAMEWORK_ENABLED` включён, но `django.contrib.sites` не
  установлен.
- `dstt.W010`: отсутствуют упакованные шаблоны или минифицированная статика.

Две команды управления (management commands) помогают диагностировать
развёртывание:

```console
python manage.py scroll_to_top_diagnose        # разрешённая конфигурация по областям, без секретов
python manage.py scroll_to_top_check_contrast  # ненулевой код, если ревизия не проходит контраст
```

## Совместимость

Официально протестированы Django 4.2 LTS, 5.0, 5.1, 5.2 LTS и 6.0 — каждая с
версиями Python, которые поддерживает соответствующий релиз Django. Поддержка
ветки Django 4 ограничивается именно 4.2 LTS, а не всей линией 4.x.

Протестированная матрица закреплена в CI (`.github/workflows/ci.yml`); `tox.ini`
отражает её для локальных запусков. В `pyproject.toml` хранится нижняя граница
поддерживаемых версий (`Django>=4.2,<7`) и классификаторы `Framework :: Django`
для протестированных релизов.

Базовые зависимости времени выполнения ограничены Django.

  работает, а что нет.

  кнопками вызова чатов, закреплённой навигацией и стеками всплывающих

  safe-area; работа при строгом CSP; наследование темы со сторонними темами
  админки; слои частичной навигации (HTMX, Turbo).

  forced-colors, reduced motion, RTL и масштабирование (200%/400%); незакрытые
